## 一、架构真实性审计：这是不是"真正的多 Agent"？

### 结论：不是。当前是 **单次 LLM 调用 + 提示词模拟多 Agent**。

```
当前实际数据流：

用户 ──POST──▶ FastAPI ──1次请求──▶ 中转站/LLM
                                        │
                                   单次生成：
                                   话题解构 + 5个Agent独白
                                   + 局长报告 + 建议 + 分层
                                        │
                              ◀──SSE 流式转发──┘
```

**证据在 `main.py` 第 153-167 行**：

```python
payload = {
    "model": req.model,
    "messages": [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_prompt},
    ],
    "max_tokens": 6000,
    "stream": True,
}
```

只有 **1 条 system message + 1 条 user message**，一次性发给 LLM。所有 Agent 的"独立思考"都是模型在同一个上下文窗口里角色扮演出来的。

### 这意味着什么？

| 维度 | 真正多 Agent | 当前方案 |
|------|-------------|---------|
| Agent 隔离性 | 物理隔离，独立上下文 | 逻辑隔离，靠提示词指令 |
| Agent 间信息泄漏 | 不可能 | 可能（模型可能让后面的 Agent 隐式参考前面的） |
| 单 Agent 失败 | 不影响其他 | 整体失败 |
| 并发能力 | 多个 Agent 并行推演 | 串行生成 |
| Token 效率 | 每个 Agent 只需自己的上下文 | 所有 Agent 共享 6000 token 上限 |
| 质量上限 | 每个 Agent 可用不同模型/温度 | 所有 Agent 同一个模型同一次生成 |

### 公平评价

提示词工程做得**相当扎实**。`system.md` 的选角算法、暗房规则、自检清单都是成熟思路。作为 MVP 来说，这个方案的 **投入产出比极高**。但它的天花板也很明确——你无法确保 Agent 之间真正没有信息泄漏，也无法让不同 Agent 用不同模型或参数。

## 二、SSE 流式传输健壮性审查

### 问题 1：同步 httpx 在异步 FastAPI 中

```python
# main.py 第 185 行
with httpx.Client(timeout=timeout) as client:   # ← 同步客户端
    with client.stream("POST", ...) as response:
```

`stream_analysis` 是一个 **同步生成器**，传给了 `StreamingResponse`。FastAPI 会把它丢进线程池执行，能工作，但：

- **每个推演请求独占一个线程**，并发 10 个用户就占 10 个线程
- 主线程的 asyncio 事件循环不受影响，但线程池默认大小有限
- 无法利用 asyncio 的协程调度优势

### 问题 2：无客户端断连检测

如果用户关闭浏览器，后端的 `httpx.Client.stream` **继续跑完全部 token**，白白消耗 API 额度。当前没有任何机制检测前端是否还在监听。

### 问题 3：无心跳机制

长推演（深挖/旗舰模式）可能 60-120 秒才出第一个 token。期间 SSE 连接无任何数据传输，代理/CDN/浏览器可能判定超时断连。

### 问题 4：无重试机制

中转站偶尔 502/503，当前行为是直接报错终止。没有重试。

### 问题 5：`max_tokens: 6000` 对旗舰模式不足

旗舰模式 + 6 个增强模块 + 8 个 Agent，完整输出很容易超过 6000 token，导致输出被截断但不报错——前端只会显示不完整的报告。

## 三、前端折叠/解析鲁棒性

### 当前实现方式

```javascript
// index.html 第 228 行
function wrapSection(root, matcher, label) {
  const headings = [...root.querySelectorAll('h1,h2,h3,h4')];
  for (const heading of headings) {
    if (!matcher.test(heading.textContent)) continue;  // ← 正则匹配标题文本
    ...
  }
}

// 调用
wrapSection(root, /独立思考暗房|暗房/i, 'Agent 各抒己见（可折叠）');
```

### 脆弱性分析

| 场景 | 会不会炸 |
|------|---------|
| 模型输出标题为 "## 独立暗房思考" | ✅ 匹配 |
| 模型输出标题为 "## 暗房发言" | ✅ 匹配 |
| 模型输出标题为 "## Agent 独立发言" | ❌ 不匹配，折叠失效 |
| 模型输出标题为 "## 各 Agent 的暗房独白" | ✅ 匹配 |
| 模型在标题中加了 emoji "## 🔒 暗房" | ✅ 匹配 |
| 模型跳过这个标题直接输出 Agent 内容 | ❌ 整个折叠功能失效 |

实测中，不同模型（GPT-4o vs Claude）对标题措辞的遵循度不同，**这个匹配是会偶尔失灵的**。

### 另一个问题：流式渲染时反复解析

```javascript
function renderOutput(live = false) {
  const root = document.createElement('div');
  root.innerHTML = marked.parse(md || '');   // ← 每个 chunk 都重新解析全文
  wrapSection(root, ...);                     // ← 每个 chunk 都重新匹配折叠
  $('oc').innerHTML = root.innerHTML + ...;   // ← 每个 chunk 都替换整个 DOM
}
```

每来一个 chunk（通常几个字），就：
1. 重新 `marked.parse` 全文
2. 重新 `querySelectorAll` 匹配标题
3. 重新替换整个 `#oc` 的 `innerHTML`

**在 4000+ 字的输出上，这会导致末段输出时肉眼可见的卡顿**，而且用户正在阅读的折叠状态（开/合）会被重置。

### Phase 2：真正的多 Agent 编排

> 目标：每个 Agent 独立请求，物理隔离，局长最后汇总

```
新架构数据流：

用户 ──POST──▶ FastAPI Orchestrator
                    │
          ┌─── 第1次调用 ───▶ LLM: 话题解构 + 动态选角
          │         │
          │    返回 Agent 列表 (JSON)
          │         │
          ├─── 并发调用 ───▶ Agent_1 独立推演 (独立 system prompt)
          ├─── 并发调用 ───▶ Agent_2 独立推演
          ├─── 并发调用 ───▶ Agent_3 独立推演
          ├─── 并发调用 ───▶ ...
          │         │
          │    收集所有 Agent 输出
          │         │
          └─── 最终调用 ───▶ 局长整合报告
                    │
          ◀── SSE 分阶段流式输出 ──┘
```

**SSE 事件类型升级**：

```
当前：status | chunk | error | done
升级：status | agent_start | agent_chunk | agent_done |
      director_start | director_chunk | director_done |
      error | done
```

**前端升级**：
```
每个 Agent 的发言实时出现在独立卡片中
局长总结在所有 Agent 完成后开始流式输出
用户可以实时看到 "Agent 🏗️ 建筑师 正在发言..."
```

**核心编排代码骨架**：

```python
async def orchestrate(req: AnalyzeRequest):
    # Phase 1: 话题解构 + 选角
    yield sse("status", label="正在解构话题")
    agents = await call_llm_for_casting(req)  # 返回结构化 JSON
    
    # Phase 2: 并发 Agent 推演
    yield sse("status", label=f"召唤 {len(agents)} 位 Agent 进入暗房")
    
    agent_tasks = []
    for agent in agents:
        task = asyncio.create_task(
            run_single_agent(agent, req.topic)
        )
        agent_tasks.append((agent, task))
    
    # 流式输出每个 Agent 的结果
    for agent, task in agent_tasks:
        yield sse("agent_start", agent=agent["code_name"])
        async for chunk in await task:
            yield sse("agent_chunk", agent=agent["code_name"], content=chunk)
        yield sse("agent_done", agent=agent["code_name"])
    
    # Phase 3: 局长整合
    all_agent_outputs = collect_outputs(agent_tasks)
    yield sse("director_start")
    async for chunk in call_director(req, agents, all_agent_outputs):
        yield sse("director_chunk", content=chunk)
    yield sse("director_done")
    yield sse("done")
```
